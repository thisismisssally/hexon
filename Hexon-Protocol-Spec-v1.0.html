1. Abstract

The Hexon Open Memory Protocol (HOMP) defines a universal standard for storing, managing, and sharing personal AI memory across any artificial intelligence system, agent, application, or device. It is an open, vendor-neutral specification that any developer, company, or individual can implement freely.

This document defines the complete protocol ‚Äî the memory data model, security architecture, API specification, transport format, and implementation requirements ‚Äî so that any product built on Hexon is fully interoperable with any other.

2. The Problem This Solves

Hundreds of millions of people use AI every single day. They pour their entire lives into these systems ‚Äî their fears, their ambitions, their health, their relationships, their most private thoughts.

And then it all disappears. Every conversation starts from zero. Every AI is an island.

FailureDescription
Memory Lock-inYour context is trapped inside each AI company's ecosystem. Switch AI ‚Äî start again.
Zero PortabilityNothing travels with you across AIs, devices, or platforms. You rebuild yourself every time.
No OwnershipYou have no control over what is stored about you, how it is used, or when it disappears.
Silo FragmentationYour Claude memory knows nothing about your ChatGPT history. Each AI has a partial picture at best.

The Hexon Protocol solves all four failures with a single open standard.

3. Design Principles

Every decision in this specification is guided by these principles, in order of priority:

PrincipleWhat It Means
User OwnershipMemory belongs entirely to the user. No implementation may store user memory on infrastructure the user does not control without explicit informed consent.
Privacy by DefaultThe most sensitive memory tiers are locked by default. No system receives sensitive data automatically.
Universal CompatibilityThe protocol must work with any AI that accepts text input. No special integration required from AI providers.
Open & Vendor-NeutralNo company owns this protocol. No implementation fee. No permission required.
Human-ReadableMemory is stored as structured JSON text. Any person can read, edit, or delete their own memory without technical expertise.
Graceful DegradationIf Hexon is unavailable, AI interactions continue normally. Hexon enhances ‚Äî it never becomes a dependency.

4. Memory Data Model

4.1 The Memory Object

Every piece of information stored in a Hexon vault is a Memory Object. The schema is:

{
  "id": "mem_a1b2c3d4",
  "created": "2026-02-27T10:00:00Z",
  "updated": "2026-02-27T18:30:00Z",
  "category": "preferences",
  "tier": "open",
  "key": "communication_style",
  "value": "Direct and concise. Dislikes waffle.",
  "source": "conversation",
  "confidence": 0.92,
  "tags": ["communication", "personality"]
}
FieldDescription
idUnique identifier. Format: mem_ followed by 8 alphanumeric characters.
created / updatedISO 8601 timestamps in UTC.
categoryOne of the standard categories defined in Section 4.2.
tierSecurity tier: open, restricted, or confidential. See Section 5.
keySnake_case descriptor of what this memory represents.
valueThe memory content as a UTF-8 string. Maximum 10,000 characters.
sourceHow this memory was created: conversation, manual, import, or api.
confidenceFloat 0.0‚Äì1.0. How confident the system is. 1.0 = user-confirmed.
tagsOptional array of strings for filtering and grouping.

4.2 Standard Memory Categories

CategoryWhat It Stores
identityName, pronouns, location, language, occupation, life stage.
preferencesCommunication style, formatting, response length, topics of interest.
personalityHumour, values, emotional patterns, working style, decision-making approach.
knowledgeProfessional expertise, skills, educational background, specialisms.
goalsShort and long-term ambitions, projects in progress, priorities.
healthMedical background, conditions, fitness context. Restricted tier by default.
financialFinancial situation and preferences. Restricted tier by default.
relationshipsImportant people, relationship context. Restricted tier by default.
historySignificant past events and experiences the user has shared.
customUser-defined categories not covered by the standard set.

5. Security Model

5.1 Memory Tiers

TierAccess Rules
openAny connected application with a valid API key may read these memories. Used for general preferences, communication style, and non-sensitive personality traits.
restrictedOnly applications explicitly granted access to this category by the user may read these memories. Access is granted per-category per-application.
confidentialNo application receives these memories automatically under any circumstances. Access requires the user to explicitly unlock for a specific session using PIN or biometric. Cannot be granted via API.

‚ö†Ô∏è Health, financial, and relationship categories MUST default to restricted or confidential tier.

‚ö†Ô∏è Implementations MUST NOT allow any API to access confidential-tier memories under any circumstances.

‚ö†Ô∏è Users may promote memories to a higher tier. They may not demote below the category default.

5.2 Encryption

Confidential-tier memories MUST be encrypted at rest using AES-256. The encryption key is derived from the user's PIN or passphrase and is never stored on the server.

Restricted-tier memories SHOULD be encrypted at rest. All data in transit MUST use TLS 1.2 or higher.

5.3 Access Log

Every API call that reads, writes, or is blocked from accessing the vault MUST be logged immutably:

{
  "timestamp": "2026-02-27T14:23:11Z",
  "app_id": "app_claude_assistant",
  "action": "read",
  "categories": ["preferences", "personality"],
  "memories_returned": 12,
  "blocked_categories": ["health"],
  "block_reason": "tier_restricted_no_permission"
}

6. API Specification

6.1 Overview

The Hexon API is a REST API served from the user's own vault server. All requests require an API key. There is no central Hexon server.

üì° Base URL: defined by the vault operator, e.g. https://vault.yourdomain.com/hexon/v1

üîë Authentication: Bearer token in Authorization header

üìÑ Format: JSON request and response bodies

6.2 Core Endpoints

GET /memories

Retrieve memories from the vault. Returns only memories the requesting application is authorised to access.

GET /memories?categories=preferences,personality&limit=50
Authorization: Bearer app_key_abc123

Response 200:
{
  "memories": [ { ...Memory Object }, ... ],
  "total": 23,
  "blocked_categories": ["health", "financial"],
  "vault_id": "vault_xyz789"
}

POST /memories

Add or update a memory in the vault.

POST /memories
{
  "category": "preferences",
  "tier": "open",
  "key": "prefers_bullet_points",
  "value": "true",
  "source": "conversation",
  "confidence": 0.85
}

GET /context

The primary endpoint for AI integration. Returns a formatted context string ready to inject into any AI conversation.

GET /context?format=text&max_tokens=2000

Response 200:
{
  "context": "The user's name is Sally. She prefers direct communication...",
  "token_estimate": 847,
  "memories_included": 18,
  "memories_excluded": 4
}

POST /connections

Register a new application with the vault. Returns an API key. Vault owner access only.

DELETE /connections/{app_id}

Revoke an application's access immediately. All future requests from that app_id return 403.

GET /log

Retrieve the access log. Vault owner access only. Supports pagination and date filtering.

7. Context Injection

7.1 How It Works

  1. User composes a message to any AI.
  2. The Hexon bridge layer intercepts the message.
  3. Bridge calls GET /context to retrieve relevant memories.
  4. Context is prepended to the user's message.
  5. The combined message is sent to the AI.
  6. After the conversation, the bridge calls POST /memories to update the vault with anything new learned.

7.2 Context Format

[END HEXON CONTEXT]

8. Implementation Requirements

8.1 Conformance Levels

LevelRequirements
CoreMemory Data Model, all three security tiers, GET /memories, POST /memories, GET /context. Minimum for Hexon compatibility.
StandardCore plus connections management, access log system. Required for production deployments.
FullStandard plus emergency trusted access, learning memory, multi-device sync, and data export/import.

8.2 MUST Requirements

All Hexon implementations MUST:

  • Enforce security tiers as defined in Section 5.1 without exception.
  • Never return confidential-tier memories via any API endpoint.
  • Log all API access to an immutable access log.
  • Store user data only on infrastructure the user controls or has explicitly chosen.
  • Support full data export in standard JSON format on user request.
  • Support full data deletion within 24 hours of user request.
  • Use TLS 1.2 or higher for all data in transit.
  • Encrypt confidential-tier memories at rest using AES-256.

8.3 MUST NOT Requirements

All Hexon implementations MUST NOT:

  • Send user memory data to any third party without explicit informed consent.
  • Use user memory data for advertising, profiling, or model training without explicit consent.
  • Prevent users from exporting, deleting, or modifying their own memories.
  • Require a central Hexon server or any Hexon-controlled infrastructure.
  • Charge for access to the protocol specification or reference implementation.

9. Learning Memory

A Hexon implementation that supports Learning Memory automatically updates the vault after AI conversations, so the system gets smarter over time without manual updates.

  1. Conversation transcript is processed by a local or cloud language model.
  2. Model extracts new facts, preferences, or updated context.
  3. Extracted memories are assigned category, tier, and confidence score.
  4. Memories with confidence above 0.7 are written to the vault automatically.
  5. Memories with confidence 0.4‚Äì0.7 are flagged for user review.
  6. Existing memories with conflicting values are updated, not duplicated.

10. Emergency Trusted Access

Allows a user to designate individuals who can access specific memory categories in defined emergency circumstances ‚Äî dementia, medical emergencies, incapacitation.

ParameterDescription
Trusted PersonA named individual with a verified contact method (email or phone).
Trigger ConditionThe circumstance under which access is granted ‚Äî defined entirely by the user.
Accessible CategoriesSpecific categories only ‚Äî never the full vault by default.
Tier LimitEmergency access never unlocks confidential-tier memories automatically.
AuditAll emergency access events are logged and the user is notified when possible.
RevocationThe user may revoke trusted access at any time.

‚ÑπÔ∏è Emergency Trusted Access is optional ‚Äî not required for Core conformance.

‚ÑπÔ∏è Must require explicit user setup ‚Äî never on by default.

‚ÑπÔ∏è Confidential-tier memories are never accessible via Emergency Trusted Access.

11. Versioning & Compatibility

The Hexon Protocol uses semantic versioning. This document defines version 1.0.

  • Major versions (2.0, 3.0) may introduce breaking changes to the API or data model.
  • Minor versions (1.1, 1.2) add new endpoints or fields without breaking existing implementations.
  • All implementations MUST include the protocol version in vault.json metadata.
  • Implementations SHOULD support reading vaults from the previous major version.

12. Developer Quick Start

Add Hexon memory support to any AI product in under one hour:

  1. Register your application with the user's vault (POST /connections). Store the returned API key securely.
  2. Before each AI call, fetch context from GET /context.
  3. Prepend the context string to the user's message using the format defined in Section 7.2.
  4. Send the combined message to the AI as normal.
  5. Optionally, after the conversation, POST any new information learned to /memories.

That is the complete integration. No AI provider agreement needed. No special SDK. No central service dependency.

13. Objections & Answers

These are the most common challenges raised about the Hexon Protocol, with honest responses.

How does context injection actually work in practice? Most AI interfaces don't let you intercept messages.
Phase 1 uses a browser extension ‚Äî one-click install, works with any AI in any browser. The extension intercepts the outgoing message, prepends the Hexon context, and sends the combined message. No AI provider needs to agree to anything. Long-term, as the protocol gains adoption, AI providers and app developers integrate the GET /context call natively ‚Äî exactly as email clients eventually built SMTP in rather than requiring users to configure it manually.
OpenAI and Anthropic are already building memory into their products. Won't they just solve this?
They will solve it for their own AI. Hexon solves it across every AI that will ever exist ‚Äî those are completely different things. More importantly, their entire business model depends on owning your data and keeping you inside their ecosystem. Building a system that gives users full ownership and portability away from them destroys their competitive moat. They have every financial incentive not to build this. And even if they did ‚Äî it would be locked to their platform. Hexon is universal and cannot be locked down by any company.
Self-hosting a server is too technical for most people.
That's exactly why there's a hosted service. One-click signup, ¬£9.99 a month, it just works ‚Äî no server, no technical knowledge required. Self-hosting is the option for technical users and privacy advocates who want full control. Most people will use the hosted service. This is identical to how Linux is free and open source, but most people happily pay for a Mac or Windows machine because they want it to just work. The protocol being open source is what creates trust ‚Äî not a requirement for everyone to run their own server.
What happens if the project dies or the maintainers disappear?
Nothing. That's the entire point of open source with an MIT License. The specification is a public document. The reference implementation is public code. If every maintainer disappeared tomorrow, anyone in the world could fork the repository and continue building. The protocol itself cannot be shut down, acquired, or killed by any company or government. This is fundamentally different from a startup that holds your data on their servers ‚Äî if that startup dies, your data is at risk. With Hexon, the worst case is the community takes over. The data always remains with the user.
AI memory is moving fast. Won't the big companies solve this in 6 months and make Hexon irrelevant?
In 6 months the big companies may solve memory within their own ecosystems. Hexon solves memory across all ecosystems, owned entirely by the user, forever. That's a different product solving a different problem. The more AI products proliferate ‚Äî and they will ‚Äî the more the fragmentation problem grows, not shrinks. Every new AI assistant, every new agent, every new tool a person adopts is another silo. Hexon becomes more valuable as the AI landscape becomes more complex, not less.
Why would AI companies or developers integrate with this? What's in it for them?
Any developer who integrates Hexon instantly offers their users a dramatically better experience ‚Äî the AI already knows the user completely from day one. That's a genuine competitive advantage. Users who have their Hexon vault set up will actively seek out products that support it, just as users seek out apps that support Sign In With Google. The network effect works in favour of adoption. Hexon also opens a direct API for developers to read and write to the user's memory ‚Äî giving them richer personalisation capabilities than they could build themselves.
